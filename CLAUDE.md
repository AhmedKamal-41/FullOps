Before starting any coding, editing, refactoring, debugging, testing, migration, scripting, configuration, or code-review task in this repository, load and follow the `plain-readable-code` skill (`.claude/skills/plain-readable-code/SKILL.md`). ## What FulfillOps is

FulfillOps is an event-driven order fulfillment and reliability platform: four independently deployable domain services (Order, Inventory, Payment, Fulfillment) coordinating through versioned Kafka events, plus a React operations console. See `README.md`, `docs/PROJECT_CHARTER.md`, `docs/ARCHITECTURE.md`, and `docs/DOMAIN_MODEL.md` for the full picture.

## Non-negotiable engineering rules

These apply to every phase from Phase 1 onward. They are restated in `.cursor/rules/fulfillops.mdc` for Cursor's final pass.

- No service may read or write another service's tables. No distributed monolith.
- No shared JPA/domain-model module. Share schemas and conventions, not persistence entities.
- Never claim exactly-once processing. Kafka delivery is at least once; correctness comes from idempotent consumers and database constraints.
- Every money, quantity, status, retry, and compensation rule belongs in a service layer or domain component, not a controller.
- Use `BigDecimal` for monetary values and integer quantities for stock.
- Use UTC `Instant` timestamps in backend contracts and persistence.
- Use UUID/ULID identifiers generated by the application. Do not expose sequential database IDs.
- All event envelopes contain `eventId`, `eventType`, `eventVersion`, `occurredAt`, `correlationId`, `causationId`, `aggregateId`, `producer`, and `payload`.
- Every retryable command endpoint accepts an idempotency key. Reusing a key with a different request payload (payload fingerprint mismatch) returns a conflict, not a false success.
- Use RFC 9457 Problem Details for HTTP errors. Never leak stack traces or secrets.
- Never store real payment-card data or real PII. The payment service is a deterministic simulator.
- Each service owns its own Flyway migrations, health/readiness endpoints, structured logs, metrics, unit tests, and Testcontainers integration tests.
- Keep the core application runnable locally with Docker Compose. Kubernetes and AWS are later packaging exercises, not development prerequisites.
- Do not add AI functionality during core phases.
- Do not commit, push, rewrite Git history, provision cloud resources, or incur cost unless the user explicitly asks.
- Do not publish throughput, latency, coverage, test-count, or scale claims until a command run in this repository actually produced them.
