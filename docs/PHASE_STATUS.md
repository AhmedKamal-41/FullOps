## Phase 3 — verification

- [x] New `contracts` Maven module (no production code) plus `contracts/events/`: `EventEnvelope.v1.schema.json`, a shared `Money.v1.schema.json`, and 11 event schemas (`OrderPlaced`, `InventoryReserved`, `InventoryRejected`, `InventoryReleased`, `PaymentAuthorized`, `PaymentDeclined`, `PaymentRefunded`, `FulfillmentAssigned`, `FulfillmentStatusChanged`, `OrderCancelled`, `OrderRequiresReview`, each `.v1`), JSON Schema 2020-12, composed via `allOf`/`$ref` rather than duplicated per file. Money is a decimal string at a fixed scale of 2; timestamps require a literal UTC `Z`; every ID field is enforced with an explicit regex pattern (not just `format: uuid`, which turned out to be annotation-only and non-enforcing by default — caught by a spike test before writing all 11 schemas, not assumed).
- [x] `EventSchemaValidationTest` (12 dynamic tests: 11 fixtures + "every schema has an example") passes — every example fixture under `contracts/events/examples/` validates against its schema, using `networknt/json-schema-validator` 3.0.6 (Jackson 3, matching Spring Boot 4.1's own default Jackson).
- [x] `contracts/README.md` states plainly that a real backward-compatibility test (does a future `v2` still accept everything `v1` did) isn't written yet because there's no second version to test against — not claimed as done.
- [x] Every service has a working transactional outbox (`OutboxEvent` entity, `OutboxEventWriter`, `OutboxRelay` — claims a batch with `SELECT ... FOR UPDATE SKIP LOCKED` in a short transaction, publishes outside any transaction, marks `PUBLISHED` only after the broker acknowledges) and idempotent inbox (`InboxEvent`, composite `(event_id, consumer_name)` key, check-then-process-then-record in one `@Transactional` method) — the pattern from ADR 0003, duplicated per service by design (no shared module), not a shared library.
- [x] No Java serialization and no JPA entity is ever put on the wire — `OutboxRelay` and `InboxEventListener` both go through a plain `EventEnvelope` record serialized to a JSON string via Kafka's own `StringSerializer`/`StringDeserializer`.
- [x] Kafka topology: one topic per service (`fulfillops.<service>.events`), keyed by order ID, `eventId`/`eventType`/`eventVersion`/`correlationId`/`causationId` also carried as headers. Retry and dead-lettering use Spring Kafka's native `@RetryableTopic` (verified current for Spring Kafka 4.1 before use) with a `NonRetryableEventProcessingException` exclusion so business rejections skip retry entirely — Resilience4j was deliberately not introduced; see ADR 0009 for the full reasoning on both.
- [x] `CorrelationIdFilter` (servlet filter, `X-Correlation-Id` in and out, MDC) on every service; `OutboxRelay` and `InboxEventListener` both set `correlationId`/`eventId` in MDC around their work. Production-like profile emits structured JSON logs via Spring Boot's built-in `logging.structured.format.console: logstash` — confirmed by actually running a service under that profile and reading real JSON log lines, not just configured and assumed.
- [x] All 8 required scenarios have a passing Testcontainers (Kafka + PostgreSQL) integration test, run for real, in every service: outbox persistence, publish success (plus correlation propagated into Kafka headers), duplicate delivery is a no-op, a processing failure rolls back the whole transaction (no inbox row left behind), a retryable failure is redelivered through a retry topic before reaching the DLT, a non-retryable failure skips retry topics entirely, DLT routing itself, and correlation propagation.
- [x] `./mvnw -B clean verify` passes for the full reactor — `contracts` plus all four services, 48 tests total (12 in `contracts`, 9 integration/security tests × 4 services), no failures, formatting clean.
- [x] No business lifecycle was implemented — no domain tables beyond Phase 2's `outbox_event`/`inbox_event`, no command endpoints, no cross-service event wiring. Each service's inbox listener currently consumes its own outbox topic as a self-contained proof of the mechanism; this is called out explicitly in ADR 0009 and `ARCHITECTURE.md` as a Phase 3 scaffold, not the real design — real cross-service listeners are a later phase's work.

Two real bugs were caught and fixed by actually running the tests, not assumed correct from the code alone, worth recording here honestly:
- A hand-rolled `KafkaTemplate`/`ConsumerFactory` bean bypassed Spring Boot's `KafkaConnectionDetails` abstraction, which is what lets the `test` profile get Kafka's address from Testcontainers automatically — fixed by removing the hand-rolled beans and configuring `spring.kafka.*` properties instead, letting Spring Boot's own auto-configuration do it correctly.
- `OutboxEvent`/`InboxEvent`'s `createdAt` field was never set in the Java constructor, so Hibernate sent an explicit `NULL` on insert that overrode the database column's `DEFAULT now()`, tripping the `NOT NULL` constraint from the Phase 2 migration — fixed by setting it in each constructor.

Also corrected mid-phase: `docs/DOMAIN_MODEL.md`'s original event catalog (from Phase 0) listed four separate `FulfillmentPicking.v1`/`Packed`/`Dispatched`/`Delivered` events plus a `FulfillmentCancelled.v1`; this phase's actual schema work consolidated those into one `FulfillmentStatusChanged.v1` (status carried as a field), and formalized `OrderCancelled.v1`/`OrderRequiresReview.v1` as real events rather than just prose descriptions of what Order Service does internally. `DOMAIN_MODEL.md` is updated to match, with the consolidation noted rather than silently rewritten.

## Phase 4 — verification

- [x] `V2__orders.sql` Flyway migration adds `orders`, `order_items`, `order_status_history`, and `idempotency_requests` to Order Service's own database only — no other service's schema touched. All public identifiers are application-generated UUIDs, never sequential DB IDs. Money columns are `NUMERIC(12,2)` with a `CHECK (... >= 0)` constraint; `order_items.quantity` has `CHECK (quantity > 0)`; `orders.status`/`order_status_history.status` are constrained by `CHECK` to the 10 statuses in `DOMAIN_MODEL.md`. `idempotency_requests` has a composite primary key `(customer_id, idempotency_key)`, which is what makes a key's ownership immutable and lets the database itself reject a concurrent duplicate — see below.
- [x] `Order` carries `@Version` for optimistic locking (not yet exercised by any status-changing code this phase, since Order Service doesn't yet consume lifecycle events — that starts in a later phase). `OrderStatusTransitions.isAllowed(from, to)` encodes the full allowed-transition table from `DOMAIN_MODEL.md` explicitly (verified by `OrderStatusTransitionsTest`: forward happy path, rejects skip-ahead and backward moves, terminal statuses allow nothing further, cancellation only before `DISPATCHED`, `REQUIRES_REVIEW` only resolves to `CANCELLED`); it is not called by any code yet, since nothing transitions order status this phase — it's the hook the inventory/payment/fulfillment consumer phases will use.
- [x] `POST /api/v1/orders` (role `CUSTOMER`, required `Idempotency-Key` header) computes totals from server-supplied SKU/quantity/unit-price line items only — the request body has no `customerId` or `totalAmount` field, so a client cannot submit either. `OrderPricing.compute` does the BigDecimal line-item and total math and rejects mixed currencies across line items (`OrderPricingTest`, 2 tests).
- [x] Order creation persists the `Order`, its `OrderItem`s, the initial `PENDING` `OrderStatusHistory` row, the `idempotency_requests` row, and an `OrderPlaced.v1` outbox event in one transaction (`OrderCreationTransaction`). Verified directly against real Testcontainers PostgreSQL via row counts, not inferred from the code (`OrderCreationIT.placingAnOrderPersistsTheOrderItemsHistoryIdempotencyRowAndOutboxEventTogether`): exactly 1 `orders` row, 2 `order_items` rows, 1 `order_status_history` row, 1 `idempotency_requests` row, 1 `outbox_event` row for that order.
- [x] Replaying the same `Idempotency-Key` with an identical payload returns the original order and creates nothing new (`OrderCreationIT.replayingTheSameKeyAndIdenticalPayloadReturnsTheOriginalOrder`: same `orderId` returned twice, still exactly 1 `orders` row). Reusing the key with a different payload is rejected as an RFC 9457 409 Conflict, and still leaves exactly 1 `orders` row for that customer (`OrderCreationIT.reusingTheKeyWithADifferentPayloadIsRejectedWithConflict`).
- [x] Concurrent identical submissions create exactly one order: `OrderConcurrencyIT` starts 10 threads that all submit the same customer, `Idempotency-Key`, and payload, releases them together via a `CountDownLatch` (each thread signals ready, then all 10 are released at once), and asserts every one of the 10 HTTP responses reports the same `orderId`, with exactly 1 row in `orders` and exactly 1 row in `idempotency_requests` afterward. This relies on the composite primary key on `idempotency_requests(customer_id, idempotency_key)`: `OrderCreationTransaction` always attempts the insert, `OrderService` (a separate, non-`@Transactional` bean, so Spring's self-invocation proxy limitation doesn't get in the way) catches the losing threads' `DataIntegrityViolationException` and re-reads the row the winning thread committed, replaying its order back to the loser instead of failing the request.
- [x] The `OrderPlaced.v1` outbox event actually produced by a real order creation validates against `contracts/events/OrderPlaced.v1.schema.json` using the same `networknt/json-schema-validator` setup from Phase 3 (`OrderCreationIT.theOrderPlacedOutboxEventValidatesAgainstItsJsonSchema`) — not just asserted to look right by eye.
- [x] `GET /api/v1/orders/{orderId}`: the owning `CUSTOMER`, or any `OPERATOR`/`ADMIN`, can read an order; anyone else gets 404 (not 403 — deliberately, so a non-owner can't distinguish "not yours" from "doesn't exist"). `GET /api/v1/orders` lists only the authenticated customer's own orders, paginated. Both authorization rules are enforced in `SecurityConfig` (URL-based `hasRole`/`hasAnyRole`) and in `OrderController`/`OrderService` (ownership check), and both layers are exercised by `OrderControllerAuthorizationTest` (10 tests: customer can place/list/read own orders; operator cannot place or list; unauthenticated gets 401; missing `Idempotency-Key` and an empty `items` list are rejected with 400; a non-owner customer gets 404 on someone else's order; an operator reading any order is treated as staff).
- [x] RFC 9457 Problem Details for every error path: `GlobalExceptionHandler` (`@RestControllerAdvice extends ResponseEntityExceptionHandler`) maps `InvalidOrderRequestException` → 400, `IdempotencyKeyConflictException` → 409, `OrderNotFoundException` → 404, Bean Validation's `ConstraintViolationException` (header/path parameters, not covered by the base class) → 400, and a generic `Exception` fallback → 500 with a fixed, non-leaking message (the real exception is logged server-side only, never returned to the caller).
- [x] `springdoc-openapi` is wired (`/v3/api-docs`, `/swagger-ui.html`, both `permitAll()`); `POST /api/v1/orders` documents the `Idempotency-Key` header, a full 201 example payload, and the 409 conflict case; the security scheme is a bearer JWT.
- [x] `./mvnw -B verify -pl services/order-service` passes: 27 unit tests (`OrderPricingTest`, `OrderServiceTest`, `OrderStatusTransitionsTest`, `WhoAmIAuthorizationTest` from Phase 2, `OrderControllerAuthorizationTest`) and 12 integration tests against real Testcontainers PostgreSQL/Kafka/Redis (`InboxEventListenerIT` and `OutboxRelayIT` from Phase 3, `OrderServiceMigrationStartupIT` from Phase 2, `OrderConcurrencyIT`, `OrderCreationIT`) — 39 tests total, 0 failures, 0 errors, formatting clean. Only `order-service` was touched this phase (its `pom.xml`, `SecurityConfig`, `CorrelationIdFilter`, and new domain/service/web code); the other three services and the shared `contracts` module were not modified, so a full-reactor rebuild was not repeated.

Two real issues were caught and fixed by actually running things, not assumed correct from the code, worth recording here honestly:
- `CorrelationIdFilter` (from Phase 3) accepted any client-supplied `X-Correlation-Id` string as-is, but `outbox_event.correlation_id` is `UUID NOT NULL` and this phase is the first time a real HTTP request path writes an outbox row — a non-UUID header value would have failed the insert. Fixed by having the filter parse the header as a UUID and fall back to a fresh random one if it's missing or not parseable, and exposing the parsed value as a request attribute so `OrderController` can pass a real `UUID` through to the transaction.
- The first version of `OrderConcurrencyIT` used `ExecutorService.invokeAll(...)`, which blocks the calling thread until every submitted task finishes — but each task was itself waiting on a `CountDownLatch` that only the calling thread (still blocked inside `invokeAll`) was supposed to release. That's a deadlock, and it hung the real build for about 24 minutes before it was caught, killed, and fixed by submitting the tasks individually with `pool.submit(...)` and collecting `Future`s instead, so the main thread's latch release isn't blocked behind the very threads waiting on it.
