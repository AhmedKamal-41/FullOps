Money columns are `NUMERIC(12,2)` with a `CHECK (... >= 0)` constraint; `order_items.quantity` has `CHECK (quantity > 0)`; `orders.status`/`order_status_history.status` are constrained by `CHECK` to the 10 statuses in `DOMAIN_MODEL.md`. `idempotency_requests` has a composite primary key `(customer_id, idempotency_key)`, which is what makes a key's ownership immutable and lets the database itself reject a concurrent duplicate — see below.

Reusing the key with a different payload is rejected as an RFC 9457 409 Conflict, and still leaves exactly 1 `orders` row for that customer (`OrderCreationIT.reusingTheKeyWithADifferentPayloadIsRejectedWithConflict`).
- [x] Concurrent identical submissions create exactly one order: `OrderConcurrencyIT` starts 10 threads that all submit the same customer, `Idempotency-Key`, and payload, releases them together via a `CountDownLatch` (each thread signals ready, then all 10 are released at once), and asserts every one of the 10 HTTP responses reports the same `orderId`, with exactly 1 row in `orders` and exactly 1 row in `idempotency_requests` afterward. This relies on the composite primary key on `idempotency_requests(customer_id, idempotency_key)`: `OrderCreationTransaction` always attempts the insert, `OrderService` (a separate, non-`@Transactional` bean, so Spring's self-invocation proxy limitation doesn't get in the way) catches the losing threads' `DataIntegrityViolationException` and re-reads the row the winning thread committed, replaying its order back to the loser instead of failing the request.

Two real issues were caught and fixed by actually running things, not assumed correct from the code, worth recording here honestly:
- `CorrelationIdFilter` (from Phase 3) accepted any client-supplied `X-Correlation-Id` string as-is, but `outbox_event.correlation_id` is `UUID NOT NULL` and this phase is the first time a real HTTP request path writes an outbox row — a non-UUID header value would have failed the insert. Fixed by having the filter parse the header as a UUID and fall back to a fresh random one if it's missing or not parseable, and exposing the parsed value as a request attribute so `OrderController` can pass a real `UUID` through to the transaction.
- The first version of `OrderConcurrencyIT` used `ExecutorService.invokeAll(...)`, which blocks the calling thread until every submitted task finishes — but each task was itself waiting on a `CountDownLatch` that only the calling thread (still blocked inside `invokeAll`) was supposed to release. That's a deadlock, and it hung the real build for about 24 minutes before it was caught, killed, and fixed by submitting the tasks individually with `pool.submit(...)` and collecting `Future`s instead, so the main thread's latch release isn't blocked behind the very threads waiting on it.

## Phase 5 — verification

- [x] `V2__inventory.sql` Flyway migration adds `product`, `stock_level`, `inventory_reservation`, `reservation_item`, `inventory_adjustment`, and `idempotency_requests` to Inventory Service's own database only. All public identifiers are application-generated UUIDs (this doc's `DOMAIN_MODEL.md` previously said `Reservation.reservationId` was a ULID — that was never true of any entity in this codebase and is now corrected to match `Order.orderId`'s established `UUID.randomUUID()` convention). `stock_level.available_quantity`/`reserved_quantity` and `reservation_item.quantity` are `INTEGER` with `CHECK >= 0` / `CHECK > 0`; `inventory_reservation.status` and `inventory_adjustment.source` are constrained by `CHECK` to their closed sets. `inventory_reservation.order_id` is `UNIQUE` — a second, database-level line of defense against ever reserving one order twice, on top of inbox dedup. Verified directly: `ReservationIT.databaseConstraintsRejectNegativeAvailableQuantityEvenBypassingTheApplication` drives a raw `UPDATE` past the application layer and confirms Postgres itself rejects it.
- [x] Concurrency strategy — documented in `ReservationTransaction`'s Javadoc and exercised by `ReservationConcurrencyIT`: every stock row is read through `StockLevelRepository.findBySkuForUpdate` (`SELECT ... FOR UPDATE`), not a plain read, so concurrent access to one SKU becomes a short queue instead of a burst of doomed compare-and-swap attempts. A multi-item reservation locks every requested SKU in a fixed order (sorted by sku) so two reservations touching overlapping SKUs can never deadlock by acquiring their locks in opposite orders. `stock_level.version` (`@Version`) and a bounded, jittered-backoff retry (`ReservationService`, `RetryBackoff`, `app.inventory.reservation.max-attempts: 20`) remain as documented defense-in-depth, not the primary mechanism — see the honesty note below for why a pure-optimistic-retry first version wasn't good enough on its own.
- [x] `ReservationTransaction.attempt` makes one all-or-nothing reservation attempt per order: every requested SKU is pre-checked against current (locked) stock inside the transaction; if any is short, nothing is mutated and a single `InventoryRejected.v1` lists every short item's `requestedQuantity`/`availableQuantity`. If every item has enough stock, every `StockLevel` is decremented, one `InventoryReservation` aggregate plus its `ReservationItem` rows are written, an `InventoryAdjustment` audit row (`source=RESERVATION`) is written per item, and exactly one `InventoryReserved.v1` outbox event is written — all in the same transaction, so a partial reservation is never observable. Verified by `ReservationIT`: full multi-item reservation, insufficient-stock-on-one-item rejection, and the atomic-rejection case where a *second*, otherwise-sufficient item is proven untouched when a sibling item in the same order is short.
- [x] `OrderPlacedListener` replaces the Phase 3 self-consuming scaffold (`InboxEventListener`, deleted, along with its test) with Inventory Service's first real cross-service consumer: `@KafkaListener(topics = "${app.messaging.order-events-topic}")` on order-service's own outbox topic (`fulfillops.order.events`), same check-then-process-then-record inbox shape and `@RetryableTopic` retry/DLT wiring as every other listener. A malformed payload is wrapped as `NonRetryableEventProcessingException` and skips straight to the DLT — a real production path, not a test-only hook; a genuine transient failure (`StockConcurrencyException`, or any other unchecked exception) goes through the retry topics first. Verified by `OrderPlacedListenerRetryIT` (mocks `ReservationService` to force each case deterministically) and `ReservationIT.duplicateDeliveryOfTheSameEventIsANoOp`.
- [x] `ReservationReleaseService`/`ReservationReleaseTransaction` implement the RESERVED → RELEASED state machine idempotently: releasing an already-released reservation is a no-op (no stock added, no duplicate audit row, no duplicate `InventoryReleased.v1`), and `InventoryReservation.version` makes two concurrent release attempts for the same order safe the same way stock locking does for reservation — verified by `ReservationReleaseIT`. Per the phase prompt ("consume future release requests only if the existing contract calls for them"), **no Kafka consumer is wired for `PaymentDeclined.v1` or a fulfillment-cancellation event this phase** — neither has a real producer yet (Payment Service is Phase 6, saga wiring is Phase 8) — so release is exposed only as a tested internal service, ready for a later phase's listener to call.
- [x] ADMIN endpoints: `POST /api/v1/products` (creates a fictional product plus its zero-stock `stock_level` row) and `PUT /api/v1/products/{sku}`; `POST /api/v1/inventory/{sku}/adjustments` (mandatory `reasonCode` from a closed `AdjustmentReasonCode` set, signed `changeQuantity`, rejects driving `available_quantity` negative). Both command endpoints require `Idempotency-Key` and reuse order-service's composite-primary-key catch-and-replay pattern, scoped per actor (JWT subject) rather than per customer. OPERATOR/ADMIN read endpoints: `GET /api/v1/inventory/{sku}` (cache-aside through Redis) and `GET /api/v1/inventory/low-stock?threshold=`. Every stock mutation — reservation, release, or admin adjustment — writes one `inventory_adjustment` audit row with actor, reason, correlation ID, before/after quantity, and timestamp; verified directly by `ReservationIT`, `ReservationReleaseIT`, and `InventoryControllerIT`. `InventoryControllerAuthorizationTest` covers the ADMIN-only vs OPERATOR/ADMIN-read role matrix.
- [x] `InventoryAvailabilityCache` is a cache-aside wrapper over Redis: `GET /api/v1/inventory/{sku}` tries Redis first, falls back to PostgreSQL on a miss and repopulates the cache; every reservation/release/adjustment evicts the affected SKUs once its transaction commits. PostgreSQL alone decides whether a reservation succeeds — `ReservationTransaction` never consults Redis. - **The same live run's diagnosis surfaced a second, related bug**: `OrderStatusTransitions` didn't allow `PENDING`, `PICKING`, or `PACKED` to transition to `REQUIRES_REVIEW`, even though `ReconciliationService.HAPPY_PATH_NONTERMINAL_STATUSES` explicitly watches all three for being stuck. `OrderRequiresReviewTransaction.markRequiresReview` silently no-ops when the transition isn't allowed (a deliberate, correct no-op for the *actually* terminal cases it was written for — already-`CANCELLED`/`DELIVERED`/already-`REQUIRES_REVIEW`) — so an order stuck in `PICKING` or `PACKED` would be "escalated" by reconciliation with an incident created, but the order status itself would never actually move, silently defeating the whole point of escalation. Caught by a new `ReconciliationServiceIT` concurrency test that seeded a stuck `PICKING` order and found it never reached `REQUIRES_REVIEW`. - `ReconciliationService`'s advisory lock originally used `JdbcTemplate` for both `pg_try_advisory_lock` and `pg_advisory_unlock` — but outside an active transaction, `JdbcTemplate` borrows a fresh pooled connection per call, and a Postgres advisory lock belongs to whichever database session (connection) acquired it. Acquiring and releasing on two different pooled connections would silently fail to release the lock, leaving it stuck held until that connection happened to close — the exact single-run guarantee this mechanism exists to provide would have been unreliable under real concurrency, the one scenario it's for. Caught before it reached a live demo: a new `ReconciliationServiceIT` concurrency test (two genuinely concurrent `reconcile()` calls) intermittently failed with neither call doing any work, traced with temporary debug logging to `tryAcquireLock` returning `false` for both callers on some runs. A transaction-scoped `pg_try_advisory_xact_lock` plus wrapping `reconcile()` in `@Transactional` was tried first and rejected — it would have made every incident/status write in one reconciliation pass share one ambient transaction, silently breaking `IncidentService`'s own established catch-`DataIntegrityViolationException`-and-recover pattern (a poisoned Postgres transaction can't run the recovery `SELECT` that pattern depends on). Fixed instead by borrowing one JDBC `Connection` directly from the `DataSource` and holding it for the lock's whole lifetime, independent of the ordinary pooled connections every other read/write in the pass still uses — verified by the same concurrency test now passing repeatedly.
